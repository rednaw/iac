# Platform Parameterization

Implementation plan for multi-platform support. Immediate driver: onboarding milledoni alongside tientjeketama.

---

## Context

The IaC currently manages one platform (tientjeketama, dev + prod). Secrets live in `secrets/infrastructure-secrets.yml` inside the IaC repo. A second platform (milledoni) needs its own server, domain, registry, and Hetzner account, operated independently by milledoni's own team.

Each org operates its own platform. rednaw operates tientjeketama; milledoni operates milledoni. No cross-platform access.

---

## Decisions

### Infrastructure secrets and config move to `.iac/` in the app repo

All platform-specific configuration moves out of the IaC repo and into a `.iac/` directory in the app repo. The IaC repo contains no secrets and no org-specific data.

`.iac/` is mounted **read/write**. IaC tasks write pub keys, SOPS config, and generated stubs directly into the app repo. The operator commits and pushes those files; task output prompts this explicitly.

```
.iac/
  iac.yml                      # SOPS encrypted credentials; image_name and app_domains unencrypted
  .env                         # SOPS encrypted production app secrets
  .sops.yaml                   # SOPS config covering both iac.yml and .env
  sops-key-<username>.pub      # one per authorized operator (generated by task)
  docker-compose.override.yml  # Traefik labels, networks, restart policies
```

`.iac/.sops.yaml` covers both `iac.yml` and `.env`. SOPS resolves it by walking up from either file; the VS Code SOPS extension finds it automatically.

`PLATFORM` and `REGISTRY_NAME` are derived from `base_domain` at runtime (e.g. `milledoni.com` → `milledoni`, `registry.milledoni.com`). The following fields are unencrypted in `iac.yml`:

| Field | Required | Description |
|---|---|---|
| `base_domain` | yes | Drives platform name, registry URL, Ansible hostnames |
| `image_name` | yes | Docker image name for the app |
| `app_domains` | yes | List of domains for Traefik TLS pre-warming |
| `server_type` | no | Hetzner server type (default: `cx23`) |

All credentials are encrypted. `server_type` falls back to the Terraform variable default when absent from `iac.yml`.

The `secrets/` directory, `sops-key-*.pub` files, and the IaC-level `.sops.yaml` are removed entirely.

### Docker Compose override for production config

The app's `docker-compose.yml` is generic — no domain, no Traefik labels, no restart policies. All production platform config lives in `.iac/docker-compose.override.yml`.

- **dev**: `docker-compose.yml` + `.devcontainer/docker-compose.override.yml`
- **prod**: `docker-compose.yml` + `.iac/docker-compose.override.yml`

A typical `.iac/docker-compose.override.yml`:

```yaml
services:
  app:
    labels:
      traefik.enable: "true"
      traefik.http.routers.app.rule: "Host(`milledoni.com`)"
      traefik.http.routers.app.entrypoints: "websecure"
      traefik.http.routers.app.tls.certresolver: "letsencrypt"
      traefik.http.services.app.loadbalancer.server.port: "3000"
    networks:
      - default
      - traefik
    restart: unless-stopped
  db:
    restart: unless-stopped

networks:
  traefik:
    external: true
```

The deploy task copies both compose files to the server and runs them together. The `traefik-dynamic-app-host.yml.j2` template reads domains from the `app_domains` list in `iac.yml` (unencrypted) to pre-warm TLS certs — avoiding fragile `Host()` label parsing.

### Terraform decoupled from SOPS

`secrets.tf` and the SOPS Terraform provider are removed. The Taskfile decrypts `app/.iac/iac.yml` before invoking Terraform and passes values as `TF_VAR_*` environment variables (not `-var` flags, to avoid process argument exposure).

Required vars: `hcloud_token`, `ssh_keys`, `allowed_ssh_ips`, `base_domain`. Optional var: `server_type` (falls back to `cx23` when absent). The Taskfile uses `// "cx23"` yq syntax so the field is not required in `iac.yml`.

`terraform validate` runs clean with no secrets file present — it checks HCL structure only.

### Task interface unchanged

The mounted `.iac/` directory implicitly defines the active platform. No `<platform>` argument is needed:

```bash
task terraform:plan  -- prod
task ansible:run     -- prod
task app:deploy      -- prod
```

Switching platforms means switching the mounted app repo via `setup-app-path.sh`.

### Terraform Cloud

One TF Cloud account per org, owned by the respective operator. `iac.yml` contains `terraform_cloud_organization` and `terraform_cloud_token`. Workspace naming: `platform-dev` / `platform-prod` within each org.

The backend organization is passed via `terraform init -backend-config="organization=..."` since Terraform backend config does not support variables.

### Hetzner

Each org has their own Hetzner account. `hcloud_token` lives in `iac.yml`. Devcontainer setup creates one hcloud CLI context from the mounted secrets.

### SOPS keys

Each operator manages their own age key. Multiple recipients: add pub key to `.iac/` and re-run `secrets:generate-sops-config`.

### App contract: one directory, two files

The top-level `app/iac.yml` is eliminated; its values are absorbed into `.iac/iac.yml`.

| Mount | Mode | Purpose |
|---|---|---|
| `.iac/` | read/write | All platform-specific config and secrets |
| `docker-compose.yml` | readonly | Generic app stack |
| `.github/workflows/build-and-push.yml` | read/write | CI workflow — written by `platform:init` |

`setup-app-path.sh` (host) validates that `docker-compose.yml` and `.iac/` directory exist — the minimum to enter bootstrap mode. Full contract validation is enforced by `devcontainer-setup.sh` at startup in operational mode.

### Devcontainer startup modes

`devcontainer-setup.sh` starts in one of two modes:

- **Bootstrap** — `app/.iac/iac.yml` does not exist. All tools available; secrets setup skipped. Message printed: `"Platform not initialised. Run: task platform:init"`
- **Operational** — `app/.iac/iac.yml` exists. Full setup: decrypt, configure hcloud/registry/TFC. Hard failure if any required file is missing or decryption fails.

### Platform bootstrapping

`task platform:init` runs inside the devcontainer and creates all remaining contract files:

1. Prompts for `base_domain`, `image_name`, `app_domains`
2. Generates age key pair → writes pub key to `app/.iac/sops-key-<username>.pub`
3. Generates `app/.iac/.sops.yaml` covering `iac.yml` and `.env`
4. Creates `app/.iac/iac.yml` with provided values and encrypted credential placeholders
5. Creates `app/.iac/.env` stub
6. Creates `app/.iac/docker-compose.override.yml` commented template
7. Writes the standard thin caller to `app/.github/workflows/build-and-push.yml`
8. Prints: fill in credentials, then `git add .iac/ .github/ && git commit && git push`

Complete sequence from scratch:

```
1. Create app repo with docker-compose.yml and empty .iac/ directory
2. Run ./scripts/setup-app-path.sh /path/to/app   (on host)
3. Open devcontainer  →  bootstrap mode
4. task platform:init
5. Fill credentials in .iac/iac.yml and .iac/.env  (SOPS extension encrypts on save)
6. Fill in domain and port in .iac/docker-compose.override.yml
7. git add .iac/ .github/ && git commit && git push
8. Reopen devcontainer  →  operational mode
```

No tools required on the host beyond Docker and the editor.

### Reusable build-and-push workflow

The IaC repo hosts `.github/workflows/_build-and-push.yml`. All app repos call it via a thin wrapper generated by `platform:init`:

```yaml
on:
  push:
    branches: [main]

jobs:
  build-and-push:
    uses: rednaw/iac/.github/workflows/_build-and-push.yml@main
    secrets:
      REGISTRY_USERNAME: ${{ secrets.REGISTRY_USERNAME }}
      REGISTRY_PASSWORD: ${{ secrets.REGISTRY_PASSWORD }}
```

The reusable workflow reads `REGISTRY_URL` and `IMAGE_NAME` from GitHub Actions variables. Logic changes once in the IaC repo; all apps benefit.

**Security:**
- `@main` with branch protection — PR review required before anything merges; no release process needed
- Explicit secrets only — no `secrets: inherit`; blast radius limited to registry credentials
- External callers can invoke the workflow but can only pass their own secrets

### CI credentials: manual, one-time

Set four values in the app repo's GitHub settings (Settings → Secrets and variables → Actions):

| Setting | Type | Source |
|---|---|---|
| `REGISTRY_URL` | variable | `registry.<base_domain>` — unencrypted in `iac.yml` |
| `IMAGE_NAME` | variable | `image_name` — unencrypted in `iac.yml` |
| `REGISTRY_USERNAME` | secret | `iac.yml` (decrypt in VS Code) |
| `REGISTRY_PASSWORD` | secret | `iac.yml` (decrypt in VS Code) |

The `gh` CLI is not used in any task automation.

### IaC CI simplified

With SOPS removed from Terraform, `SOPS_AGE_KEY` and the `sops-setup` action are removed from the IaC CI pipeline. `terraform validate` runs without secrets.

## Implementation phases

The work is split so each phase leaves the system working. Steps 1 and 2 touch only the IaC repo and are independently mergeable today. Step 3 is the only coordinated change across two repos.

### Phase 1 — IaC repo only, pure addition/removal (no risk)
- Remove `SOPS_AGE_KEY` and `sops-setup` from `run-iac-tests.yml`
- Add `_build-and-push.yml` reusable workflow

### Phase 2 — IaC repo only, Terraform decoupled from SOPS
- Remove `secrets.tf` and SOPS provider from `versions.tf`
- Add Terraform input variables
- Taskfile: decrypt `secrets/infrastructure-secrets.yml` (current path) and export `TF_VAR_*`
- Secrets stay where they are; when secrets move in Phase 3 it is a one-line path change

### Phase 3 — Coordinated: IaC repo + tientjeketama app repo
- App repo: create `.iac/` with all files; extract Traefik labels into `.iac/docker-compose.override.yml`; strip them from `docker-compose.yml`
- IaC repo: update `devcontainer.json` mounts, `setup-app-path.sh`, `devcontainer-setup.sh` (both modes), Taskfile paths, deploy task to run both compose files, remove `secrets/` directory
- Must land together — the devcontainer is broken between the two commits

### Phase 4 — IaC repo, Ansible (requires Phase 3 deployed)
- Hostname convention `<platform>-<env>` throughout inventory
- `traefik-dynamic-app-host.yml.j2` reads `app_domains` from `iac.yml`
- `fail2ban.yml` renames `rednaw.conf` → `platform.conf`
- All depend on `base_domain` being available from `iac.yml`

### Phase 5 — IaC repo, new platform support (skipped)
- `task platform:init` — not implemented
- Placeholder `.iac/` directory in the IaC repo for new operators
- Milledoni onboarding

### Phase 6 — Docs (done)
- Updated `new-project.md`, `application-deployment.md`, `secrets.md`, `traefik.md` for `.iac/` contract and override pattern
- Updated `joining.md`, `launch-devcontainer.md`, `README.md`, `monitoring.md`, `remote-ssh.md`, `troubleshooting.md`, `registry.md`, `private.md` for `app/.iac/iac.yml` and CI credentials
- Removed platform-parameterization backlog item

## Implementation detail

### 1. Secrets and config migration (tientjeketama app repo)
- Move `secrets/infrastructure-secrets.yml` → `.iac/iac.yml`; add `image_name` and `app_domains` as unencrypted fields
- Move `app/.env` → `.iac/.env`
- Move `secrets/sops-key-*.pub` → `.iac/`
- Generate `.iac/.sops.yaml` covering `iac.yml` and `.env`
- Create `.iac/docker-compose.override.yml` with Traefik labels, networks, restart policies
- Remove Traefik labels, network declarations, and `restart` from `docker-compose.yml`
- Remove `secrets/` from the IaC repo and the IaC-level `.sops.yaml`

### 2. devcontainer.json
- Replace all individual file mounts with one read/write `.iac/` mount
- Keep `docker-compose.yml` as readonly mount
- Add `build-and-push.yml` as read/write mount
- Remove the old `iac.yml` and `.sops.yaml` mounts

### 3. setup-app-path.sh
- Validate `docker-compose.yml` exists and `.iac/` directory exists (bootstrap gate)
- Remove checks for individual files previously validated on the host

### 4. devcontainer-setup.sh
- Decrypt `app/.iac/iac.yml`; derive `PLATFORM` and `REGISTRY` from `base_domain`
- Write one hcloud context, one registry auth entry, one TFC token
- Bootstrap mode: skip secrets setup, print init message
- Operational mode: hard failure if any required file is missing or decryption fails

### 5. Terraform
- Remove `secrets.tf` and the SOPS provider from `versions.tf`
- Update `locals.tf` to read from `var.*`
- Add variables: `hcloud_token`, `ssh_keys`, `allowed_ssh_ips`, `base_domain`, `server_type`
- Terraform tasks: decrypt `app/.iac/iac.yml`, export `TF_VAR_*` (`server_type` uses `// "cx23"` yq fallback), pass `--backend-config="organization=..."` on init

### 6. Tasks
- `secrets:keygen`: write pub key to `app/.iac/sops-key-<username>.pub`
- `secrets:generate-sops-config`: read `app/.iac/sops-key-*.pub`, write `app/.iac/.sops.yaml`
- Update all task preconditions from `iac.yml` / `app/.env` paths to `app/.iac/iac.yml` / `app/.iac/.env`
- Add `platform:init` task
- Remove `app/iac.yml` placeholder; add `app/.iac/` placeholder directory
- Update `Taskfile.yml` help text

### 7. Ansible
- Hostname convention: `<platform>-<env>` throughout inventory, derived from `base_domain`
- `traefik-dynamic-app-host.yml.j2`: read domains from `app_domains` list in `iac.yml`
- `fail2ban.yml`: rename deployed config from `rednaw.conf` to `platform.conf`
- Deploy role: copy both `docker-compose.yml` and `.iac/docker-compose.override.yml` to server

### 8. Reusable workflow
- Add `.github/workflows/_build-and-push.yml` to the IaC repo

### 9. IaC CI
- Remove `SOPS_AGE_KEY` secret from the IaC GitHub repo
- Remove `sops-setup` action from `run-iac-tests.yml`

### 10. Docs
- Update `docs/new-project.md` for the `.iac/` contract and bootstrapping sequence
- Update `docs/application-deployment.md` — remove `iac.yml`, document override pattern
- Update `docs/secrets.md` — `.iac/` directory and key management
- Update `docs/traefik.md` — labels now in `.iac/docker-compose.override.yml`
- Add CI credentials setup to onboarding docs
- Remove the backlog item
